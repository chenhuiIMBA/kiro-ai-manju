## 元教训（跨领域复用）

### M1. 规则的约束对象必须明确区分

**模式**：执行一条规则前，先确认它约束的是哪个**层面**：

- **构图/视觉层面**：画面中如何呈现 → 影响 prompt 写法和景别选择
- **传参/配置层面**：传什么给 API/脚本 → 影响调用参数
- **命名/组织层面**：文件怎么命名/归类 → 影响目录结构
- **审计/结构化层面**：登记在表格里是什么 → 表格可能是正文的**子集**而非**全集**

**历史案例**（见 `memory/2026-05-08-snapshot.md`）：
- 教训 5：规则 4"单 seg 最多 2 角色同框"误解为"ref-images 只传 2 个"（层面混淆：构图 vs 传参）
- 教训 10：`chars` 列表遗漏背景角色（层面混淆：叙事焦点 vs 画面可辨识）
- 教训 16：seg 切分对白覆盖只查汇总表（层面混淆：结构化表格 vs 正文全集）

**适用**：任何多层级规则体系。踩一次同类错再想"之前好像遇到过" = 正是这条元教训要防的。

### M2. 全局约束必须下沉到执行时的 checklist

**模式**：全局文档里写的"设计原则"不是执行者的"必查项"。AI 在执行具体阶段时只看该阶段的 checklist，不会主动回顾全局。任何全局约束必须**在每个相关阶段的 checklist 里显式列出**才能生效。

**历史案例**（见 `memory/2026-05-08-snapshot.md` 教训 14）：ASSETS-INDEX 同步规则原本只在 `_convention.md` 设计原则段，EP03 新建廊下场景后被漏更新，直到升级为门禁 5 + 下沉到 5 个阶段验收清单才彻底解决。

**适用**：任何"全局规则 × 分阶段执行"的工作流设计（CI/CD 的多 stage、多 agent 协作同理）。

### M3. 根因治理优于人工对冲

**模式**：当某类错误反复出现（如"又忘了双写"、"又忘了更新索引"），不要加一条"更严格的检查规则"让人工对冲。要么：

- **工具化**：把两份数据源改成单源派生（`render_assets.py` 从 seg-config.py 单源渲染 assets.md，而不是要求"双写"）
- **重命名**：把容易误解的字段名改得语义明确（`chars` → `visible_chars`，`audio` → `speaking_chars`）
- **校验脚本**：把人工检查改成机械扫描（`validate_script.py` 检查正文 vs 汇总表一致性）

**历史案例**：
- 教训 15 "双写规则" → 升级为 `render_assets.py` 单源渲染
- 教训 10 字段命名 → 升级为 `visible_chars` / `speaking_chars`
- 教训 14/16 → 升级为 `regen_assets_index.py` / `validate_script.py`

**适用**：任何"让 AI/人工记得做某事"的弱规则。能用工具消除的错误不要靠意志力。

### A1. prompt 描述不能因"防模型行为风险"而删除画面元素（2026-05-09）

**问题**：EP03 seg-4 v1 模型把 prompt 里"沈鹿溪方向"的字样当成对白素材，自动生成女主报姓名对白。我为了防止 v2/v3 再次触发，**把沈鹿溪从 prompt 描述里完全删掉**——结果 v3 画面中沈鹿溪消失，违背剧本分镜。

**根因**：元教训 M1 的具体形态。两层约束混淆：

- **画面内容层**（剧本/分镜决定）：sq画面里应当出现什么 → prompt 必须完整描述
- **模型行为层**（API 特性决定）：prompt 文本可能被跨模态误用（如对白生成/字幕生成）→ 通过**显式约束声明**处理

这两层是**正交关系**，不是对立关系。防行为风险的正确做法是"加约束声明"（如"无对白无台词无人声"），**不是"删画面描述"**。删画面描述直接违反了 M1 的另一条——"ref-images 不能替代 prompt 描述"，因为参考图只提供外观锚点，不决定画面里是否**存在**某个角色。

**修复**：seg-4 v4 恢复沈鹿溪完整描述 + 保留"无对白"声明 + 加入 shenluxi multiview。

**防御规则**：组装/修改 prompt 时遇到"是否删除某角色/动作描述"的抉择，先问：
1. 剧本/分镜里这个元素**应当可见**吗？是 → 不能删，必须写入 prompt 正文
2. 担心的风险（对白/字幕/水印等）是**跨模态生成行为**吗？是 → 加显式约束声明（"无对白""无字幕""无水印"）
3. 两步独立处理，不允许用步骤 2 的解药去治步骤 1 的病

**适用**：任何"prompt 文本可能被模型跨模态解释"的生成系统（视频/图片/音频）。本质是 M1 的"约束对象必须明确区分"——画面内容约束 ≠ 模型行为约束。

### A2. 场景库资产只含环境布局，不含角色圣经里的主要角色（2026-05-09）

**问题**：EP03 seg-4 v5 方案需要一张"集体跪拜布局"的场景变体图。我的首版 prompt 里把"殿门口边缘小桃探头偷看"写进了变体图——**小桃是角色圣经里的主要角色**（有 `04-character/xiaotao/` 个人库）。用户指出后发现这混淆了场景库和角色库两层资产的抽象边界。

**根因**：元教训 M1 的具体形态。两层资产的责任边界混淆：

- **场景库**（`03-scenes/{场景}/`）：长期可复用的**空间+群演环境布局**资产
- **角色库**（`04-character/{角色}/`）：角色圣经里每个主要角色的**外观+音色**资产

主要角色是动态入场的（谁在哪集哪 seg 出现由剧本决定），如果把主要角色烙进场景图，该场景图就只能用于"该角色在场"的剧情，失去跨集/跨剧情的复用价值；还会和 Seedance ref-images 里的角色 multiview 产生语义冲突（两个"小桃"打架）。

**修复规则（直接定义场景库资产边界）**：

`03-scenes/{场景}/` 下的 `main.png` / `{变体}.png` 允许包含：
- 建筑空间（立柱、穹顶、地面、家具、屏风）
- 光线、色调、材质、时段氛围
- **群演布局**——没有角色圣经个体条目的群体布景（如请安时两列嫔妃、宫宴时列班大臣、街市路人），仅作为**环境的一部分**，不渲染面部细节

严禁包含：
- 角色圣经（`04-character/{角色}/`）里任何主要角色（有 front.png/multiview/voice-ref 三件套的）
- 主要角色的辨识特征（特定面部/特定服饰编号）

主要角色必须由 Seedance 的 ref-images（multiview）+ prompt 文字描述联合注入。

**判断标准**：在 `04-character/` 下有目录的 → 主要角色，不能进场景图；没有的 → 群演，可作为环境布局进场景图。

**适用**：所有"场景×角色"分层的创作流程。通用原则：**可复用抽象资产不应嵌入易变具象元素**。

### A3. 起始画面是软约束，景别跨度大时会失效，需要运镜过渡（2026-05-09）

**问题**：EP03 seg-4 v5 使用 seg-3 v3 尾帧（中近景仰拍沈鹿溪）作为起始画面，但 prompt 开头直接是"远景+正面+殿内全景"。模型无法从中近景特写直接切成远景全景，于是**自由运镜**，运镜过程中把沈鹿溪朝向从"面朝高台"改成"面朝大门"——起始画面看似"没生效"。

**根因**：seedance 的"多模态参考生视频"模式下（我们默认模式），起始画面是 `reference_image`，属于**软约束**。API 文档原话："可通过提示词指定参考图片作为首帧……若需严格保障首尾帧一致，优先使用图生视频-首尾帧（配置 role 为 first_frame/last_frame）"。

**防御规则**：seg 组装时加入硬自检项——**起始画面景别 ≈ prompt 0-Ns 景别**：
- 匹配（同景别或相邻景别）→ 模型可平滑衔接
- 不匹配（特写 → 全景等跨 ≥2 级景别）→ **必须在 0-Ns 节拍加入运镜过渡**（如"镜头从上一 seg 的特写缓慢向后拉远并俯降到全景平视"）
- 严禁跨景别硬切（模型会自由运镜重决构图，连带朝向/位置都会漂）

**已验证对策**：seg-4 v6 采用"0-3s 承接起始画面（中近景仰拍）缓慢拉远俯降→3-5s 到达远景静止"的运镜节奏，朝向正确。

**适用**：所有使用多模态参考模式+起始画面图的视频生成。本质是"软约束下模型会利用自由度做它自己认为合理的事，除非你在运动过程中也约束住"。

### A4. kiro executeBash 长命令 stdout 不可靠，API 调用用 controlBashProcess+tee（2026-05-09）

**问题**：3 次 Seedream 调用（长 prompt 变量+多个 --ref-images 路径+长输出）的 stdout 在 kiro executeBash 返回结果里被截断/消失，我误以为调用失败连续重试 3 次——实际都成功了，浪费 35.6K tokens 和 3 次任务配额。

**根因**：kiro executeBash 对长命令的 stdout 捕获不稳定，尤其是"命令本身长 + 输出包含大量信息"的组合。不是 shell/API 问题，是 kiro 执行环境层的问题。

**防御规则**：所有 Seedance/Seedream/长运行 API 调用**一律用 controlBashProcess**（不用 executeBash），并在命令里加 `2>&1 | tee <log-file>`：
```bash
python3 video-generation/scripts/seedance.py create --prompt "..." --ref-images ... 2>&1 | tee projects/<project>/.tmp_<action>.log
```
任务 ID 从日志文件读取（`readFile` 或 `cat`），不从命令 stdout 读取。

**适用**：任何在 kiro 环境中执行的"长命令+长输出+慢完成"组合。是工具层约束，不是规则层约束。

### A5. 审核失败必须先追问具体原因再提新版本（2026-05-09）

**问题**：EP03 seg-4 v4 用户说"审核不通过"就直接跳到场景变体图方案，没问具体失败点。后来追溯才发现 v4 的实际问题不明——盲目切换方案浪费 519K tokens（视频延长模式的一次完整生成）。

**根因**：我把"审核不通过"当作"换方案"的触发器，而不是"需要诊断"的信号。少了"问清楚哪里不对"这一步，新方案就是赌博。

**防御规则**：任何版本审核失败后，**未追问具体失败点前不得提交新版本**。追问格式：
```
v{N} 审核失败的具体原因方向是？
- 人物朝向/位置错
- 景别/构图错
- 角色外观漂
- 对白/音频错
- 布局漂移
- 其他（请描述）
```
拿到明确反馈后再组装 v{N+1}，针对性修复而不是重新试方案。

**适用**：任何"多版本迭代"的生成任务。本质是：失败信号要解析为具体诊断，不是触发器。

### A6. 场景变体图是同场景多 seg 布局一致性的首选方案（2026-05-09）

**问题**：同场景多 seg 在不同机位/时刻需要**相同的群演布局**（如请安时两列嫔妃跪拜），如果只传 `main.png`（空场景）+ prompt 文字描述，模型对"两列纵队"这种几何关系理解不稳定，经常漂移成"几人并肩一排"或"三三两两散布"。

**根因**：ref-images 的角色 multiview 只提供外观，不提供群体几何关系；prompt 文字描述对空间布局的信息量有限。

**三种方案对比**：

| 方案 | tokens/seg | 跨 seg 一致性 | 资产复用性 | 推荐度 |
|------|-----------|-------------|-----------|--------|
| A. 纯 prompt 文字描述 | ~217K | 低（容易漂）| — | ❌ |
| B. 视频延长（reference_video）| ~519K | 高（视频锚定）| — | 特殊场景 |
| C. 场景变体图（本方案）| ~217K + 17.8K/变体 | 高（布局图锚定）| **可多 seg 多集复用** | ✅ **默认首选** |

**使用规范**：
- 命名：`{场景}/{布局名}.png`（如 `xuanxiu-dadian/jibai-buju.png`）
- 生成：`seedream create -i main.png`（图生图，显式声明"空间/材质/色调与参考图完全一致"）
- **严禁内容**：角色圣经（`04-character/`）里的任何主要角色，只能有群演
- 使用时机：同场景 ≥2 个 seg 需要固定群演布局

**适用**：AI 漫剧工作流。跨项目同类需求（大场景多机位）也适用。

### A7. 情境转换必须有剧情触发因，不得凭空切换（2026-05-10）

**问题**：EP03 seg-6 原本只有萧珩"朕知道了"一句对白就结束，紧接 seg-7 的"嫔妃散场+温婉走来"。结果 seg-7 生成反复出现"嫔妃倒退走"、"向龙椅走去"等错向，v1→v6 共迭代 6 次。根因并非 prompt 方向描述问题，而是**剧本层面缺少散场触发因**——嫔妃为什么突然散场？没有对白/动作告诉她们"可以走了"，模型只能凭空脑补散场动机，动机不清时方向就漂。

加入"回宫吧"散场宣告到 seg-6 后，seg-7 的散场就有了剧情因果：皇帝宣告→嫔妃起身→陆续离开。模型理解触发→动作的因果链，方向自然对。

**根因**：情境转换（群体状态切换、场景切换、情绪基调切换）必须有**剧情触发因**（trigger）。触发因形式：
- **对白宣告**（散场、宣旨、命令、问候）
- **标志性动作**（某人起身、某人落座、某人离场）
- **事件节点**（铃响、门开、时辰到）

触发因必须**在情境转换发生**时存在于画面/对白中，不能只在剧本备注里。模型是视觉+对白驱动，剧本备注"请安结束"对它不可见。

**防御规则**（固化到 05-script.md 验收 + 07-video.md 切分自检）：

阶段 05（剧本层）自检项：
- [ ] **情境转换触发因覆盖**：扫描剧本正文中所有"请安散场/进场/落座/起身/门开/铃响"等情境切换点，每一处必须有对应的对白宣告或关键动作作为触发因，且触发因在切换发生前 1-3s 出现

阶段 07（seg 切分层）自检项：
- [ ] **相邻 seg 的情境转换有触发因**：如 seg-N 末是"静态某情境"，seg-(N+1) 是"变化后情境"，前后衔接处必须有触发因。触发因可放在 seg-N 末或 seg-(N+1) 首，视叙事节奏决定

**历史案例**：EP03 seg-6 → seg-7（散场无触发因，补加"回宫吧"后解决）

**适用**：所有"群体状态转换 + 场景/场面切换"的剧本创作。单人单场景的情感推进不受此规则约束，因为情感转换的触发因通常是内部（潜台词），画面表现靠微表情足够，不需要外部触发。

### A8. 群体运动方向：动作动词优于空间位置词（2026-05-10）

**问题**：EP03 seg-7 v2-v5 反复失败。v2 写"嫔妃朝殿深处退出"，模型把"殿深处"理解为"龙椅方向"（大殿内部的深处），结果嫔妃朝龙椅走；v5 写"向画面左右两侧散开"，模型理解成"原地分散"（散开 ≠ 散场）；v6 改为"嫔妃们陆续离开大殿"后一次通过。

**根因**：元教训 M1 的新形态——"建筑空间语义" vs "画面坐标语义" vs "行为语义" 三者混淆：

- **建筑空间词**（殿深处/殿门/廊下/宫深处）：模型按建筑学常识解读，常与画面坐标不一致（大殿的"深处"可能是模型理解的"内部"而非"外部"）
- **画面坐标词**（画面左右两侧/画面中央/画面前景）：模型按画面构图理解，但对"出画/入画"这种**运动方向**理解不稳定（"散开"可解读为"原地分散"或"向外扩散"）
- **行为语义词**（离开大殿/走出画面/走向沈鹿溪）：模型理解动作意图，方向自然正确

**优先级**：描述群体/个体**运动方向**时，**动作动词优于空间位置词**：

| 场景 | ❌ 空间词 | ✅ 动作动词 |
|------|---------|----------|
| 嫔妃散场 | "朝殿深处退出" / "向两侧散开" | "陆续离开大殿" / "走出画面" |
| 角色进场 | "从殿门方向入画" | "走进大殿" / "从远处朝镜头走来" |
| 角色离场 | "朝宫深处走去" | "转身走向宫深处" / "背对镜头走开" |
| 角色靠近 | "从远景中线走来" | "朝沈鹿溪走去" / "走向她身旁" |

**保留使用空间位置词的场景**：描述**静态构图**（谁在画面哪里）时仍用空间词，如"沈鹿溪在画面左侧"、"萧珩位于画面中央"。

**防御规则**（已固化到 07-video.md prompt 书写建议）：写 prompt 时区分：
- **静态位置** = 画面坐标词（左/中/右、前景/背景）
- **运动方向** = 动作动词（走向X / 离开X / 靠近X），配合可选的"朝画面X侧"辅助

**适用**：任何涉及群体运动、角色走位的视频生成 prompt。本质是 M1 的"约束对象必须明确区分"——静态位置和运动方向需要不同的描述语法。

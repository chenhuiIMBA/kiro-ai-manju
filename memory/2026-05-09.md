## 4. 经验教训

### 做得好的

1. **及时识别结构性问题**：seg-4 v2 布局漂移时没有盲目重试，而是分析出"ref-images + prompt 文字"对布局锚定不足，引出变体图路径
2. **用户审核反馈转化为教训沉淀**：prompt 删姓名、场景图含主要角色、景别不匹配三类问题都写入了 MEMORY.md 活跃教训
3. **资产安全规则执行到位**：每次弃用版本都先备份再替换，没有发生资产丢失

### 可改进的（按影响程度排序）

#### C1. 审核失败时必须追问具体原因（执行纪律）

**问题**：seg-4 v4 用户说"审核不通过"就直接跳到新方案（场景变体图），没问"v4 哪里不对"。后来追溯才发现当时的具体问题不明——**信息缺失下的方案切换浪费了 519K tokens**。

**防御规则**：任何版本审核失败后，未追问具体问题前**不得提交新版本**。必须先用户确认失败原因方向（人物朝向/景别/对白/外观/布局/其他），再组装新 prompt。

#### C2. 景别匹配必须作为 prompt 自检硬项（能力升级）

**问题**：seg-4 v5 起始画面（seg-3 v3 中近景仰拍）与 v5 的 0-5s（远景全景）景别不匹配，模型无法平滑衔接 → 自由运镜 → 朝向重决 → 出错。

**根因**：多模态参考生视频模式下，起始画面是**软约束**；如果 prompt 开头景别与起始画面不一致，模型必须"硬切 or 自由运镜"，两种都不可控。

**防御规则**：seg 组装时新增硬自检项——"**起始画面景别 ≈ prompt 0-Ns 景别**"。如果必须跨景别（特写 → 全景），**必须在 0-Ns 加入运镜过渡节拍**（如"镜头缓慢拉远"），**不得硬切**。

#### C3. kiro executeBash 长命令输出截断（工具层问题）

**问题**：3 次 Seedream 调用（带长 prompt 变量+多个 --ref-images 路径）的 stdout 被截断，前两次看起来没输出，我以为调用失败重试，实际都成功了——重复调用浪费 35K tokens。

**根因**：kiro 的 executeBash 对长命令/长输出有截断机制，不稳定。

**防御规则**：所有 Seedance/Seedream 调用（命令长+输出长+耗时长）**一律用 controlBashProcess + tee 到文件**，**不用 executeBash**。读日志文件确认任务 ID 而不是读 stdout。

#### C4. 门禁执行纪律：原子化意识（执行纪律，已识别但仍反复）

**问题**：Seedream 变体图生成后漏登记 assets.md 和 ASSETS-INDEX。这是教训 #14 / MEMORY M2 的同源问题，固化到每阶段验收清单但在执行中仍被拆成"先生成再补登记"。

**防御规则**：下次生成任何资产前，先明确原子操作清单（生成→重命名→登记 assets.md→更新 ASSETS-INDEX→清理临时文件），**在同一个回复内全部完成**。不允许以"下一步收尾"为由拆分。

#### C5. 场景库变体图作为新工作流能力（能力升级）

**问题**：`ai-manju/stages/06-scenes.md` 目前只定义了 main.png / multiview.png。**变体图**（同场景不同布局）没有模式。本次试用验证了它的价值——同场景多 seg 的布局一致性难题，变体图是比视频延长（2.4x 成本）更高性价比的解决方案。

**防御规则**：考虑把 **场景变体图** 作为 06-scenes.md 的正式子模式，明确：
- 命名规范：`{场景}/{布局名}.png`（如 `xuanxiu-dadian/jibai-buju.png`）
- 生成方式：`seedream create -i main.png`（图生图，锚定空间/色调/材质）
- **严禁内容**：角色圣经里的主要角色（只能有群演布局）
- **使用时机**：同场景 ≥2 个 seg 需要固定群演布局时

#### C6. 方案推荐的自一致性（表达纪律）

**问题**：第一次提"场景图变体 vs 视频延长"方案对比时，表格里标"B（推荐）"后面又说"推荐 C"，被用户指出。

**根因**：我在方案推导过程中更新了结论，但没回头同步修改表格中的标注。

**防御规则**：方案对比表格的"(推荐)"标注，在提交消息前做最终核对，确保和正文结论一致。

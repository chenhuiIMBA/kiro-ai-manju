# AI 漫剧工作流 - 分步详解

> 与 `stages/` 目录下各阶段文件一一对应的详细指南。详细命令和模板见阶段文件本身，此文件侧重**跨阶段的宏观流程和设计考虑**。

## 术语速查

| 术语 | 含义 |
|------|------|
| beat | 骨架里的剧情节拍（episode-skeleton 中每集 3-5 个 beat） |
| seg | 视频片段（`seg{N}.mp4`） |

## 阶段 01：整季叙事大纲

详见 `stages/01-outline.md`。

### 设计考虑

**三者合并审核（大纲 + 角色圣经 + 骨架）**：以前分三次确认，每次之间有等待时间。但三者本身有紧耦合——角色 Want/Need 决定人物弧，人物弧驱动分集结构。拆开审核的结果往往是"确认 A 之后做 B 时发现 A 需要改"。现在要求一次性交付，用户审核一次。

**成本估算前置**：整季开工前给用户预算区间。20 集 AI 漫剧含迭代和跨集审计约 500-1200 元 API 成本，用户有心理预期后续决策不慌。

**全局扫描 + narrative-state 初始化一次完成**：骨架写完做全局逻辑扫描时，扫描本身需要提取所有信息揭露点和伏笔——正好同步初始化 narrative-state，避免重复扫描。

## 阶段 02：风格板（Moodboard）

详见 `stages/02-moodboard.md`。

### 设计考虑

**禁止角色形象**：Moodboard 发生在角色设计之前。如果此阶段生成带具体角色的图，后续角色设计会受这张图的暗示影响，导致角色外观被 Moodboard 先入为主。只允许剪影/背影/远景人影。

**变体轮换机制**：Seedream 对同样的"日漫"关键词会产出高度相似的光影和构图。通过主动注入色彩策略、构图、光影氛围的变体描述，避免 5 张图看起来像一张图。

---

## 阶段 03：灯光哲学 + 配色方案

详见 `stages/03-lighting.md`。

### 设计考虑

**灯光在角色之前**：灯光是视觉骨架，决定角色在什么光里被看见。如果先设计角色再定灯光，灯光约束会被角色 multiview 的"均匀柔光"稀释。反过来，先定灯光 → 角色设计可以针对"会穿暗场景"的角色设计深色服装、配发型光方案。

**语义标记而非像素色**：AI 无法精确匹配像素级颜色，但可以通过一致的语义标记（`night_ambient`、`warm_key`）减少漂移。这是配色方案的工程化本质——用命名约束替代数值约束。

**日漫/写实美学分野**：分离光策略（是否始终使用轮廓光）是日漫 vs 写实风格最核心的分野。这一选择决定后续所有角色从背景浮出的方式。

## 阶段 04：角色设计

详见 `stages/04-character.md`。

### 设计考虑

**三步走：front → multiview → voice-ref**：
- front（全身正面图）锁定外观金标准
- multiview（多视图组合）用图生图生成，保证面部和服装跨视角一致，这张图传入 Seedance
- voice-ref 建立角色音色 DNA，防止配音跨集漂移

**不出表情变体**：情绪表达由 prompt 文字 + 场景光影驱动，不在 Seedream 阶段做表情图。表情图的维度爆炸（喜/怒/哀/惊 × 各角色）成本不划算。

**差异化锚点**：20 集里每个角色要独特。面容特征对撞表强制任意两个角色的脸型/眼型/眉型/鼻型/嘴型/下颌至少 3 项不同——不靠服装颜色区分，靠骨相区分。

**角色先于场景**：2026-05-06 阶段重排后角色设计前置。理由见阶段 06 "按需生成"段。

## 阶段 05：剧本（单集）

详见 `stages/05-script.md`。

### 设计考虑

**剧本的职责边界**：剧本写镜头**语义**（远景/推镜/切到），不切 seg 边界。seg 边界由阶段 07 根据技术规则（时长、角色同框数、场景跳转）切分。这种分离让剧本专注于叙事，技术约束单独处理。

**加载 narrative-state 而非所有前置剧本**：第 15 集写剧本不需要读前面 14 集的完整剧本（tokens 爆炸）。narrative-state 已经浓缩了关键信息——已揭露信息、角色当前态度、未兑现伏笔、上一集摘要。需要具体台词原文时按需回溯。

**四模板强制选型**：钩子 → 布设 → 对抗 → 高潮 → 揭示，这是"微三幕"模板。其他三个模板（反转/提问回答/对比）各有适用场景。强制选型比自由发挥更可靠——新手不选就按 A，老手按需切换。

**narrative-state 硬门禁**：剧本确认后立即更新 narrative-state，否则不进视频阶段。这条规则是为了防止"先写 5 集剧本再回头补 state"——那时候已经忘记细节，补出来的 state 是错的。

## 阶段 06：场景图（按需生成）

详见 `stages/06-scenes.md`。

### 设计考虑

**按需生成替代一次性穷举**：旧流程（04-scenes 提前出齐全部场景）在剧本还没写完时就猜所有场景，导致很多场景生成了却没被用上，部分剧本中出现的新场景又缺失。现在阶段 05 剧本确认后按该集需要处理场景图（复用/修改/新建）。

**场景库替代故事板**：每集每段都生成关键帧（故事板）成本高、一致性差、返工多。现在所有主场景存在 `03-scenes/{场景名}/` 里，跨集复用同一张 main.png。色调漂移降低 80%。

**场景图只用 main.png，不要 multiview**：教训 #9。场景的核心价值是色调+材质+空间氛围，一张高质量 main.png 足够传递给 Seedance。视频中的运镜变化靠 prompt 文字描述控制，不靠场景参考图的多视角。角色 multiview 保留（角色需要正/侧/背外观一致性）。

**变体命名机制**：同一场景的白天/夜晚/宴席等变体用 `main-{变体名}.png` 命名，用图生图生成，强制"空间布局不变 + 变化点明确"。

## 阶段 07：视频 Prompt 组装 + 依赖链并行生成

详见 `stages/07-video.md`。

### 设计考虑

**依赖链内串行、链间并行**：这是本工作流最大的性能优化。以前要求所有 seg 逐个串行提交，一集 8 seg × 3 分钟等待 = 24 分钟。现在：
- 只有**起始画面衔接的 seg**（起始画面来自上一 seg 尾帧）必须等前置完成
- 不同场景的 seg 无依赖，可并发提交
- 控制并发 ≤ 3 避免 API 限流

一集 8 seg 按依赖 DAG 拆成 3 条并行链，总时长从 24 分钟降到 8-10 分钟。

**ref-images 逻辑槽位 + 物理索引**：以前 ref-images 按"实际传入顺序"编号，导致同一张角色多视图在不同 seg 的"图片N"编号不一致。但 Seedance 不支持稀疏槽位——没法"空占位"。现在明确区分：
- 逻辑槽位（场景/角色A/角色B/起始画面/补充）= assets.md 里的概念，顺序固定
- 物理编号（图片1、图片2、...）= 实际传入的顺序，缺失槽位跳过
- prompt 里用物理编号，AI 组装时根据本 seg 实际 ref-images 动态计算
- assets.md 里记录逻辑→物理的映射表，便于人工核对

> "槽位-起始画面" 这个名字从**消费者（当前 seg）**的角度看。它承担的是当前 seg 的"首帧"角色。虽然文件名叫 `lastframe_seg{N-1}.png`（上一 seg 的尾帧），但在当前 seg 的 ref-images 语境下，它的逻辑角色是起始画面。

**prompt 版本化而非覆盖**：精修时经常需要改 prompt。旧脚本用 sed 覆盖——历史 prompt 丢失，无法对比"为什么 v2 比 v1 好"。现在版本历史表 + (vN, 历史) 代码块，所有版本可追溯。

**10 条切分规则**：从实战总结的 Seedance 2.0 弱项：3+ 角色、场景跳转残影、复合运镜、回忆闪回、对白启动延迟等。遵守这些规则一次通过率明显上升。

### DAG 并行示例

一集 8 seg 的依赖图：

```
链1（食肆）：seg-1 (文生) → seg-2 → seg-3
链2（大殿）：seg-4 (文生) → seg-5 → seg-6
链3（闪回）：seg-7 (文生)
链4（结尾）：seg-8 (文生)
```

4 条链可并发。实际并发限 ≤ 3，所以链 1-3 同时启，链 4 等其中任一空出来。

## 阶段 08：合成

详见 `stages/08-composite.md`。

### 设计考虑

**纯 concat，不加转场**：短剧节奏快，硬切比转场效果更紧凑。转场（淡入淡出、翻页等）在 AI 漫剧里显得廉价。

**字幕后期烧录**：Seedance 生成的字幕经常是乱码或错别字。字幕在 ffmpeg 阶段独立烧录，错别字改 SRT 重烧即可，不需要重跑视频。

**自动备份**：`composite.sh` 在覆盖 `ep{NN}-raw.mp4` 或 `ep{NN}-final.mp4` 前自动备份到 `06-output/_backup/`。遵守 asset-safety 规则。

## 阶段 09：成片检查

详见 `stages/09-review.md`。

### 设计考虑

**AI 预筛 + 用户复核**：以前宣称"AI 自检画面、用户只审音频"——过于乐观。实际 AI 对光影连续性、色调漂移这类问题的判断能力不如用户直觉。现在：
- AI 高置信维度（画面崩坏、字幕、叙事）→ AI 自主判断
- AI 中置信维度（场景色调、角色外观）→ AI 预筛疑似问题帧 → 用户复核
- AI 低置信维度（光影连续、口型）→ 用户主导

预筛的价值是**缩小用户需要关注的范围**，而不是替代用户审核。

**分级修复权限**：字幕错别字不需要用户同意，改 SRT 重烧即可。Seedance 视频重生成必须用户同意——既避免失控消耗 API，也避免漂移失控。

## 阶段 10：批量生产

详见 `stages/10-batch.md`。

### 设计考虑

**信心度分级**：20+ 确认点会拖慢整个流程。按集数分 P0/P1/P2 三级，逐级授权 AI 自主。但任何一集出现漂移立即降级——信心度是浮动值，不是单向解锁。

**剧本追溯修改决策树**：写到第 15 集发现第 2 集伏笔有问题，有三条路：
- 改当前和之后（默认）
- 改历史剧本文档，视频保持原样
- 改历史剧本 + 重生成视频（核弹级，仅关键错误）

决策原则：剧本的弹性 > 完美一致性。能用"后续集圆回来"解决的不重做视频。

**每 5 集跨集审计**：视觉审计对比角色面部特写帧；声音审计用第 1 集 golden 片段作为金标准。AI 无法做声音审计，必须人工听。

## 阶段 11：发布前处理（可选）

详见 `stages/11-publish.md`。

### 设计考虑

**多平台转码**：不同分发平台对码率、时长、分辨率要求不同。一次合成，多次转码。

**封面帧自动提取**：从 3 秒钩子提取封面。好的钩子 = 好的封面。

**发布文案模板化**：集标题 + 100 字简介 + 标签 + 下集预告。AI 根据剧本和 narrative-state 自动生成。

## 常见工作流问题

### 卡在哪个阶段怎么办

看 `STATE.md` 的 `当前阶段` 字段 → 读对应的 `stages/NN-*.md` → 按文件内指令继续。

### 换会话如何恢复

1. 读 `STATE.md`（当前阶段、当前集、信心度）
2. 读 `01-scripts/narrative-state.md`（当前叙事状态）
3. 读 `05-videos/ep{当前集}/assets.md`（当前集已生成的 seg）
4. 继续从当前阶段推进

### 某段视频崩了怎么办

1. 检查问题类型（见阶段 09 的问题分类表）
2. 字幕问题 → 改 SRT 重烧
3. 视频问题 → 先报告用户，用户同意后 `regen_seg.sh`
4. 场景整体偏差 → 重生成场景库 multiview，再重生成受影响的 seg

### 跨集漂移严重怎么办

1. 立即停止批量生产
2. 降级到 P0 全确认
3. 运行跨集审计（视觉 + 声音）
4. 对比 EP01 golden 与当前集：是外观漂移还是声音漂移？
5. 外观漂移 → 检查 multiview 是否被改、恢复到初版
6. 声音漂移 → 检查 voice-ref 是否一致、必要时切到 B 方案（TTS 逐句合成）
7. 审计通过后升级到 P1 继续

# 阶段文件规范

每个 `NN-description.md` 是 AI 漫剧工作流的一个自包含执行单元。

## 术语表（强制遵守 🆕）

为避免混淆，整个工作流使用以下固定术语：

| 术语 | 含义 | 禁用同义词 |
|------|------|-----------|
| **beat** | 骨架里的剧情节拍（episode-skeleton.md 中每集的 beat1..beatN，N 通常 3-5） | 不叫 seg、段、片段 |
| **seg** | 视频片段（Seedance 生成的 `seg{N}.mp4`） | 不叫 beat、节拍、镜头、shot |
| **起始画面** | 视频段首帧（通过 ref-images 传入 + prompt 声明）。在 ref-images 槽位里叫"槽位-起始画面"——从当前 seg 的消费者角度命名，物理文件来源是上一 seg 的尾帧 | 不叫 first-frame（是被禁用的 API 参数）、不叫槽位-尾帧 |
| **尾帧** | 视频段末帧（`lastframe_seg{N}.png`） | 不叫 last frame、末帧、结尾帧 |
| **voice-ref** | 角色音色参考（`voice-ref.mp3`） | 不叫 音色样本、配音参考、voice sample |
| **金标准** | 角色/场景的主参考图（character front.png / scene main.png） | 不叫 主图、基准图 |
| **multiview** | 多视图组合图（传入 Seedance 的 ref-images） | 不叫 多视角图、组合图 |
| **narrative-state** | 叙事状态追踪文件（`narrative-state.md`） | 不叫 剧情状态、story state |
| **skeleton** | 分集骨架（`episode-skeleton.md`，每集 3-5 beat） | 不叫 大纲、分集纲要（那是 series-bible） |
| **片段依赖链** | 同场景连续 seg 通过起始画面衔接（前一 seg 的尾帧文件作为后一 seg 的起始画面）形成的链 | 不叫 段链、序列链 |

## 阶段编号约定（🆕）

11 个阶段连续编号，无跳级：

```
01-outline      02-moodboard    03-lighting     04-character
05-script       06-scenes       07-video        08-composite
09-review       10-batch        11-publish (可选)
```

> ⚠️ **阶段重排说明（2026-05-06）**：原流程为 `04-scenes → 04-character → 06-script`。现改为 `04-character → 05-script → 06-scenes`。原因：场景图需结合剧本内容按需生成，而非提前猜测一次性生成全部场景。旧项目升级时需同步更新 `STATE.md` 的阶段进度列表。

## 四大门禁（全局硬性约束 🆕）

所有阶段必须遵守这四条门禁，违反任何一条 = 阻断执行：

### 门禁 1：Prompt 审核门禁

**调用任何外部生成 API 之前，必须将完整 prompt + 全部参数发送给用户审核，确认后才可提交。**

适用所有生成工具：Seedance、Seedream、TTS。

执行顺序：组装 prompt → 发用户审核 → 用户确认 → 提交 API。

**预授权豁免**（P2 等级）：
用户可以**预先授权**（通过信心度等级 P2）AI 在特定阶段自主组装 prompt 并提交。豁免范围：
- 适用阶段：阶段 07 视频 prompt 组装（P2 阶段）
- 不适用阶段：大纲、骨架、角色设计、场景库、重生成（这些永远需要审核）
- 豁免的是"逐次发审核"动作，**不豁免** assets.md 完整记录 prompt 的义务
- 豁免可随时撤销：用户发现漂移即降级到 P1，恢复每次审核

> 即 P2 = 预授权批处理，不是"跳过审核"。所有 prompt 仍必须完整记录到 assets.md，用户可事后审计。

### 门禁 2：即时登记门禁

每生成一个资产文件（图片/视频/音频），必须立即完成以下三步后才能生成下一个：
1. **重命名**：将生成的文件重命名为规范文件名
2. **登记 assets.md**：将完整 Prompt + 命令参数 + 文件名写入 assets.md
3. **确认写入成功**：检查 assets.md 已保存

**禁止**：批量生成后补录 / 攒一批再统一重命名 / 写"待补录"占位符。

**原因**：assets.md 是跨会话恢复的唯一锚点。会话中断时，未登记的资产 = 丢失 Prompt 记录 = 无法复现。

### 门禁 3：STATE.md 门禁

每个阶段完成后，必须立即更新 STATE.md（勾选已完成阶段、更新当前阶段、更新最后更新时间）。不更新 STATE.md 不能进入下一阶段。STATE.md 是新会话恢复项目进度的唯一入口。

### 门禁 4：narrative-state 门禁

每集剧本（阶段 06）确认后，必须立即更新 `narrative-state.md`，否则不进视频生成阶段（阶段 07）。

## 设计原则

1. **自包含**：阶段文件不引用其他阶段文件。所有 prompt 模板、命令、约束写在本文件内。
2. **命令可复制**：`bash` 代码块中的路径用相对路径（以项目根为基准），可直接执行。
3. **显式验收**：每个阶段以 checklist 结束，逐项核对后手动勾选。
4. **跨会话可追溯**：每个生成阶段必须在 `assets.md` 中记录**完整 Prompt 原文 + 命令参数**。URL 会过期、图片可能丢失，但 Prompt 是唯一的可复现锚点。
5. **目录命名**：所有目录和文件使用**英文或拼音**，禁止中文。`04-character/guyuan/` ✅ | `04-character/顾鸢/` ❌
6. **资产速查表**：项目根目录的 `ASSETS-INDEX.md` 是所有生成资产的快速索引（中文名 ↔ 目录名 ↔ 文件状态）。每次新建/修改/删除资产后**同步更新**此文件。AI 进入任何阶段前先读此文件定位资产。
7. **路径占位符**：

| 占位符 | 含义 | 解析规则 |
|--------|------|----------|
| `<video-gen>` | video-generation skill 目录 | 优先工作区根目录下的 `video-generation/`；否则 `/root/.openclaw/workspace/skills/video-generation` |
| `<tts-skill>` | tts skill 目录 | 优先工作区根目录下的 `tts/`；否则 `/root/.openclaw/workspace/skills/tts` |
| `<ai-manju>` | 当前 skill 目录 | 优先工作区根目录下的 `ai-manju/`；否则 `/root/.openclaw/workspace/skills/ai-manju` |

> ⚠️ 执行命令前确认占位符已替换为当前环境的实际路径。

## assets.md 规范（强制 🆕）

每个有内容产出的阶段，必须在对应目录写入 `assets.md`。

### 基础格式

```markdown
# {阶段/角色/场景} 资产清单

## 生成 Prompt

| 文件 | 完整 Prompt | 命令参数 | 任务 ID | 生成时间 |
|------|------------|---------|---------|----------|
| file_01.png | "完整的 prompt 原文..." | --size 9:16 --seq auto | task_xxx | 2026-05-05T21:28 |

## 产出文件

| 文件 | 来源 | 下载 URL |
|------|------|----------|
| file_01.png | Seedream 文生图 | https://... |
```

**规则**：
- Prompt 列必须是发送给生成工具的完整原文，不是摘要、不是关键词、不是"场景描述"
- 命令参数（`--size`、`--ratio`、`--duration`、`--seq` 等）一并记录
- URL 可能过期，但 Prompt 不会

### 视频 seg 的标准格式（阶段 07 专用，强制）

`05-videos/ep{NN}/assets.md` 中每个 seg 必须使用以下标准块格式（`regen_seg.sh` 依赖此格式解析）：

```markdown
## 片段划分表

| seg | 时长 | 状态 | task_id | seed | 首帧来源 | 依赖 | 备注 |
|-----|------|------|---------|------|---------|------|------|
| 1   | 10s  | completed | cgt-xxx | 123456 | 文生视频 | — | seg1.mp4 |
| 2   | 8s   | completed | cgt-yyy | 789012 | seg-1 尾帧 | seg-1 | seg2.mp4 |
| 3   | 12s  | pending | — | — | 文生视频 | — | |

## 段详情

### seg-1

**时长**：10s
**场景**：{场景名}
**首帧来源**：文生视频
**依赖**：—
**画面概述**：{一句话}

#### Prompt (v1, 当前)

\`\`\`text
{完整 Seedance prompt 原文}
\`\`\`

#### ref-images（逻辑槽位 → 物理编号）

| 逻辑槽位 | 物理编号 | 文件路径 |
|---------|---------|----------|
| 槽位-场景 | 图片1 | `./03-scenes/{场景名}/main.png` |
| 槽位-角色A | 图片2 | `./04-character/{角色A}/multiview.png` |
| 槽位-角色B | 图片3 | `./04-character/{角色B}/multiview.png` |
| 槽位-起始画面 | 图片4 | `./05-videos/ep{NN}/lastframe_seg{N-1}.png` |
| 槽位-补充 | 图片5 | `./03-scenes/_props/hand.png`（手部） |
| 槽位-补充 | — | —（本段无其他补充） |

> ⚠️ 物理编号按实际传入顺序（场景→角色A→角色B→起始画面→补充）计算，跳过缺失槽位。Prompt 正文里的"图片N"必须用物理编号。
>
> **命名说明**："槽位-起始画面" 从当前 seg 的消费者角度命名（这张图在当前 seg 里承担"首帧"的角色）。文件本身是上一 seg 的 `lastframe_seg{N-1}.png`。
>
> **补充槽位可多行**：手部/道具/换装可同时存在，每张占一个物理编号。总 ref-images 张数 ≤6 最佳。

#### audio

- 音频1: `./04-character/{角色A}/voice-ref.mp3`
- 音频2: `./04-character/{角色B}/voice-ref.mp3`

> ⚠️ **纯动作段（无对白/独白/画外音）省略此区段**，记作"无（无人声段）"。`--audio` 只在需要 TTS 生成人声时传入对应角色的 voice-ref；环境音效由 `--generate-audio true` 和 prompt 中的 `<环境音>` 标注独立生成，不依赖 voice-ref。

#### 版本历史

| 版本 | task_id | seed | 时间 | 说明 |
|------|---------|------|------|------|
| v1 | cgt-xxx | 123456 | 2026-05-05 | 初版 |

### seg-2

...
```

**版本历史规则**：
- 首次生成为 v1
- 每次用 `regen_seg.sh` 重跑后追加 v2/v3，不覆盖旧版本
- 当前使用的版本标注 `(当前)`
- Prompt 若不变则只加版本表行；Prompt 变化则新增一个 `#### Prompt (v2, 当前)` 块，旧版改为 `#### Prompt (v1, 历史)`

**ref-images 逻辑槽位规则**：
- 传入顺序固定：场景 → 角色A → 角色B → 起始画面 → 补充
- 缺失的槽位跳过，**不占物理编号**（Seedance 不支持稀疏编号）
- "图片N" 在 prompt 正文中指代 = 实际传入的物理顺序
- assets.md 必须同时记录逻辑槽位和物理编号，便于人工核对 prompt 里的引用
- "槽位-起始画面" 从消费者（当前 seg）角度命名。文件物理来源是上一 seg 的 `lastframe_seg{N-1}.png`

## narrative-state.md 规范（强制 🆕）

叙事状态追踪文件，存放于 `01-scripts/narrative-state.md`。每集剧本完成后必须更新。
这是 AI 编写后续集剧本时的核心上下文输入——替代"读取前面所有集完整剧本"的低效方式。

```markdown
# 叙事状态追踪 — {剧名}

**最后更新集**: EP{NN}

## 已揭露信息（观众已知）

| 集 | 揭露内容 |
|----|---------|
| EP01 | {观众在这集获得的关键信息} |
| EP02 | ... |

## 未揭露信息（按计划揭露集数）

| 计划集 | 内容 | 状态 |
|--------|------|------|
| EP12 | {待揭露的信息} | pending |
| EP18 | ... | pending |

## 角色当前状态

### {角色名}
- **对{角色B}的态度变化轨迹**：{EP01-04 警惕} → {EP05-09 信任} → {EP10起 怀疑}
- **Want 进度**：{百分比}（{当前阶段描述}）
- **Flaw 松动程度**：{未松动 / 开始质疑 / 已突破}
- **最后关键台词**：EP{NN}-seg{N} "{台词}"
- **最后关键动作**：EP{NN}-seg{N} {动作描述}

## 未兑现的伏笔

| 埋设位置 | 伏笔内容 | 计划兑现 | 状态 |
|----------|---------|---------|------|
| EP02-beat3 | {伏笔描述} | EP14 | pending |
| EP06-beat2 | {伏笔描述} | EP11 | pending |

## 上一集摘要（EP{NN}）

- beat1: {一句话}
- beat2: {一句话}
- beat3: {一句话}
- ...（按实际 beat 数量列出）
- 最后一 beat: {一句话} — 结尾悬念：{描述}

```

**更新规则**：
- 每集剧本确认后立即更新，不允许跳集补写
- "已揭露信息"只增不删——观众知道的不能收回
- "未揭露信息"兑现后移入"已揭露"并标注实际集数
- "未兑现伏笔"兑现后标记 `done`，超过计划集数 3 集未兑现标记 `overdue`
- 角色态度变化轨迹必须用 `→` 连接，标注集数范围
- 上一集摘要每次覆盖写（只保留最近一集）

**追溯修改例外**（与 10-batch 的方案 3 配合）：
- 当用户执行追溯修改方案 3（改历史剧本 + 重生成视频）时，原"已揭露信息"可能被修改
- **不删除**原条目，改为标记 `corrected-by-EP{NN}` 并追加新版本条目
- 格式示例：
  ```
  | EP02 | ~~女主是老板女儿~~ → corrected-by-EP15 |
  | EP02 | 女主是孤儿（追溯修改，EP15 执行） |
  ```
- 未兑现伏笔被追溯废弃同理，标记 `deprecated-by-EP{NN}` 而非删除

## episode-skeleton.md 规范（强制 🆕）

分集骨架文件，存放于 `01-scripts/episode-skeleton.md`。整季大纲确认后编写，是所有集剧本展开的基础。

### beat 数量（3-5 浮动）

每集 beat 数量由剧情复杂度决定，不再固定 4：

| beat 数 | 适用场景 |
|--------|---------|
| 3 beat | 紧凑单场景集（纯对峙、纯告白、单一空间内完整事件） |
| 4 beat | 默认选择（经典微三幕结构） |
| 5 beat | 复杂多场景集（含转场、多视角、双线并进） |

> 上限硬定 5——超过 5 beat 的集大概率剧情散乱，应拆分或删节。
> 下限硬定 3——少于 3 beat 难以构成完整戏剧弧线。

### 文件格式

```markdown
# {剧名} 分集骨架

## EP01: {集标题}（4 beat）
- beat1: {一句话描述钩子}
- beat2: {一句话}
- beat3: {一句话}
- beat4: {一句话} — 悬念：{钩子}

## EP02: {集标题}（3 beat）
- beat1: ...
- beat2: ...
- beat3: ... — 悬念：...

## EP03: {集标题}（5 beat）
- beat1: ...
- beat2: ...
- beat3: ...
- beat4: ...
- beat5: ... — 悬念：...
```

**编写规则**：
- 每集标题后用 `（N beat）` 声明本集 beat 数量
- 分批生成（每次 5 集），每批带着前面已自检通过的骨架作为上下文。分批是 AI 内部推进单位，用户审核一次性进行
- 每 beat 一句话能说清楚即可（不设硬字数上限，一般在 20-50 字内），只写"发生什么"，不写镜头语言
- 每集**最后一 beat** 必须标注悬念钩子
- 骨架确认后不轻易改动——改骨架意味着后续所有集的连锁调整
- 如需调整，必须同步检查 narrative-state 中的伏笔和信息揭露计划
- beat 数量选择依据写在每集标题旁的括号里（用户审核时判断是否合理）

## 用户确认门禁分级（🆕）

20+ 确认点会拖慢整个流程。根据项目进展分级：

| 等级 | 适用阶段 | 确认点 | 门禁 1 状态 |
|------|---------|--------|-----------|
| P0 全确认 | 项目初期（第 1 集）及核心资产（大纲、骨架、角色、场景库） | 每个 prompt 发用户审核，每个资产用户确认 | 严格执行 |
| P1 批次确认 | 第 2-5 集 | 同类资产批次合并审核（如 5 张场景主视角一次看完） | 严格执行（批次审核） |
| P2 预授权批处理 | 第 6 集起 | 阶段 07 视频 prompt 预授权 AI 自主组装提交，用户仅审成片 | **预授权豁免**（仅限阶段 07 视频 prompt；assets.md 仍必须完整记录） |

**升级触发条件**：连续 3 集通过 P1 审核无重大返工 → 可升级到 P2。
**降级触发条件**：任一集出现风格/角色漂移 → 降回 P1，补做跨集审计。降级后门禁 1 恢复严格执行。

> ⚠️ P2 ≠ 跳过门禁。P2 只豁免"逐次发送审核"的动作，仍需：
> - 完整 prompt 记录到 assets.md（事后审计凭证）

## 版本化规则（🆕）

### 成片版本化

`06-output/ep{NN}-final.mp4` 每次重新合成时，旧版本必须备份：
- 新生成：`ep{NN}-final.mp4`（当前）
- 旧版本：`06-output/_backup/ep{NN}-final-v{N}-{timestamp}.mp4`

`composite.sh` 自动处理。

### seg 版本化

视频段重新生成时（`regen_seg.sh`），旧文件备份到 `05-videos/ep{NN}/_backup/`，assets.md 中的版本历史表新增一行，prompt 如有变更另存为 v2 块。

### 资产备份规则（补充 asset-safety）

删除或替换任何项目资产文件前，必须先备份到：
- 资产所在目录的 `_backup/` 子目录（以时间戳命名）
- 或 `07-consistency-check/_backup/`（跨阶段备份）

# 阶段 01：整季叙事大纲

> 工具：DeepSeek（纯文本推理，不需要图片/视频模型）

## 前置条件

- 项目已初始化，`STATE.md` 存在

## 输入

- 用户偏好（题材、角色方向、画风、集数、目标受众）

## 流程

### 步骤 0：创意选题（🆕 如用户无明确方向）

> 用户可能只有模糊偏好（"甜宠"、"女性向"），没有具体故事。此步骤通过"创意漏斗"帮用户从模糊偏好收敛到具体题材。
> 如果用户已有明确故事方向，跳过此步骤直接进入步骤 1。

**创意漏斗流程**：

```
第 1 轮：类型推荐
  → 根据用户偏好（受众、题材方向），推荐 3-5 个爆款类型
  → 用户选定一个类型（如"霸总甜宠"）

第 2 轮：题材细化
  → 在选定类型下，给出 5-7 个具体题材方案
  → 每个方案包含：一句话概念 + 核心冲突 + 主角数 + 集数
  → 用户选定一个题材

第 3 轮：确认参数
  → 确认画风、对话呈现、目标受众等创作参数
  → 可由 AI 根据题材和受众推荐默认值
```

**类型推荐参考维度**：
- 目标受众的消费习惯（女频/男频/通吃）
- 当前市场验证过的爆款类型（重生复仇、霸总甜宠、悬疑反转、战神归来等）
- 适合 60 秒 × 多集格式的叙事结构（每集需要悬念钩子）
- AI 漫剧的视觉优势（场景切换少的题材比场景密集的题材更容易保持一致性）

**题材方案格式**：
```
{编号}.「{标题}」（{子类型}）
{一句话概念描述}。{核心冲突}。{主角数}主角，{集数}集。
```

### 步骤 1：收集创意输入

向用户询问：
- 故事题材（玄幻/都市/悬疑/甜宠/其他）
- 核心角色数量和基本设定方向
- 画风偏好（日漫/韩漫/国漫/写实/插画）
- 对话呈现风格：字幕模式 / 气泡台词模式 / 混合
- 每集时长（默认 60 秒）
- 总集数（如 20 集）
- 目标受众

### 步骤 2：画风细化（强烈推荐）

> "日漫"不是一种风格——90年代赛璐珞和2010s京都动画的视觉基因完全不同。细化画风大幅降低后续迭代次数。

| 维度 | 可选方向 | 示例 |
|------|---------|------|
| 年代参照 | 80s赛璐珞 / 90s经典 / 2000s数字 / 2010s现代 / 当代 | 2010s 现代数字动画 |
| 作品/工作室参照 | 京都动画风 / 新海诚风 / 吉卜力风 / MAPPA 风 | 新海诚风——极致光影、照片级背景 |
| 线条特征 | 粗重外轮廓 / 纤细均一 / 有笔触变化 / 无线条厚涂 | 纤细均一、干净利落 |
| 上色方式 | 硬边平涂 / 软边渐变 / 厚涂 / 水彩渗化 | 软边渐变、柔和过渡 |
| 阴影风格 | 分层色块式 / 柔和模糊 / 硬边阴影 / 网点纸 | 分层色块式、2-3 层明暗 |

将选定特征写入 `03-storyboard/visual-style-spec.md`。

### 步骤 3：生成整季大纲

产出 `01-scripts/series-bible.md`，内容结构：

```markdown
# {剧名} 整季大纲

## 主题
具体、可争论。不是"爱与正义"，而是"偏执的复仇最终吞噬复仇者本身"。

## 中心戏剧问题
驱动全剧的核心问题。每集都在回答这个问题的一小部分。
例："林小夜能在找到凶手的同时不失去自己的人性吗？"

## 系列大弧
主角从{起点}到{终点}的完整旅程，要体现主题。

## 人物弧
详见 `character-bible.md`。此处仅列出各角色的弧线摘要：

### {角色A}
- 起始状态：{性格、处境}
- 成长路径：{遭遇什么、如何改变}
- 终局状态：{性格、处境}
（Want/Need/Flaw/Ghost 详见角色圣经，不在此重复）

## 角色关系图
标注：信赖/敌对/暗恋/利用/保护/竞争/师徒等

## 分集大纲
初始版（步骤 3）：
| 集数 | 标题 | 核心事件 | 悬念钩子 | 主题推进 |
|------|------|---------|----------|---------|
| 01 | ... | ... | ... | ... |

骨架完成后补充列（步骤 6 全局检查时一并完成）：
| 集 | 标题 | 核心事件 | 信息揭露 | 角色弧位置 | 集末状态变化 | 悬念钩子 |

## 冲突主线
- A 故事线（主线）：...
- B 故事线（支线/感情线）：...
```

### 步骤 4：用户确认

提交整季大纲，确认后进入步骤 5。

### 步骤 5：生成角色圣经（与步骤 3 同批确认亦可）

产出 `01-scripts/character-bible.md`（角色内在画像）：

```markdown
## {角色名}
### 外在
- 外貌：{描述}
- 服装：{描述}
- 标志性动作/物品：{咬嘴唇/转戒指/永远戴着帽子}

### 内在（四维）
- **Want**：{表面欲望}
- **Need**：{深层需求。关键：Want ≠ Need}
- **Flaw**：{核心缺陷/错误信念}
- **Ghost**：{创伤事件}

### 语言指纹
- 节奏：{快速/缓慢/时快时慢}
- 句长：{短句/长句/混合}
- 态度：{直接/迂回/讽刺/自我贬低/居高临下}
- 词汇特征：{文雅/粗俗/专业术语/网络用语}
- 口头禅或习惯句式：{...}
- 对白时的小动作：{摸耳朵/看地面/手不停比划}
```

### 步骤 6：生成分段骨架（episode-skeleton）

> ⚠️ 这是 20 集 × 4 段 = 80 个段落的全局叙事骨架。每段一句话，是后续逐集展开剧本的基础。
> 没有骨架就展开剧本 = 盲写，到第 10 集必然散架。

产出 `01-scripts/episode-skeleton.md`。

**编写方式**：分批生成，每次 5 集（20 段），每批带着前面已确认的骨架作为上下文。

```
第 1 批：EP01-EP05（用户确认后）
第 2 批：EP06-EP10（带 EP01-05 骨架作为上下文）
第 3 批：EP11-EP15
第 4 批：EP16-EP20
```

**格式**：

```markdown
# {剧名} 分段骨架

## EP01: {集标题}
- seg1: {一句话，≤30字，只写发生什么}
- seg2: {一句话}
- seg3: {一句话}
- seg4: {一句话} — 悬念：{钩子}

## EP02: {集标题}
...
```

**质量检查**（每批确认前）：
- 信息揭露节奏是否均匀（有没有连续 3 段以上在"聊天"而无新信息）
- 悬念钩子是否每集 seg4 都有
- 角色弧线是否平滑推进（对照 series-bible 的人物弧）
- 有没有重复的冲突模式（连续多集用同一种冲突类型）

**全局逻辑一致性检查 + 叙事状态初始化**（🆕 全部批次完成后，一次性完成）：

> ⚠️ 分批检查容易漏掉跨批次的逻辑问题（如第 2 集埋的细节在第 15 集使用，分属不同批次）。
> 全部 80 段写完后，做一次完整的全局扫描。扫描过程中同时提取信息，产出 narrative-state.md 初始版和分集大纲补充列，避免重复扫描。

**逻辑检查**（逐条过）：
- [ ] **信息归属正确**：每条信息揭露的主语明确——是观众知道了，还是角色 A 知道了？不能混淆
- [ ] **角色认知边界**：每集中角色的行为是否只基于该角色当前已知的信息？
- [ ] **身份/人物指代一致**：涉及多个人物时，每集对同一人物的指代是否一致？
- [ ] **因果链完整**：列出所有悬念钩子，逐条检查后续是否有解答或推进。无解答的标记为遗漏
- [ ] **情感锚点闭环**：列出所有道具/台词/动作锚点，检查首次建立→中间强化→最终回收的完整链路。后续使用但前面没有铺垫的标记为遗漏
- [ ] **时间线无矛盾**：事件发生的先后顺序是否自洽？
- [ ] **动机连贯**：反派/配角的行为动机在整季中是否一致？
- [ ] **配角线收束**：所有在前半段出场的配角，在后半段是否有回响或收束？不能出场后消失

**同步产出**（扫描过程中一并完成，不需要额外扫描）：
1. **补充分集大纲**：在 `series-bible.md` 的分集大纲表格中补充"信息揭露"和"角色弧位置"列
2. **初始化 narrative-state.md**：从骨架中提取未揭露信息计划表、伏笔清单，初始化角色状态

> narrative-state.md 在每集剧本完成后更新（阶段 05 负责），此处只做初始化。格式见 `_convention.md` 的 narrative-state.md 规范。

### 步骤 7：用户最终确认

提交全部产出物（大纲 + 角色圣经 + 骨架 + 叙事状态初始版 + 成本估算），用户确认后进入视觉基建阶段。

### 附：整季成本粗估（🆕）

> 在用户最终确认前，提供一个 API 调用成本的粗略估算，避免跑到一半预算不够。

**估算公式**（按 {总集数} 集 × 4 段计算）：

| 阶段 | 调用次数 | 单次成本（约） | 小计 |
|------|---------|--------------|------|
| Moodboard（Seedream） | 5-8 张 | ~0.1 元/张 | ~0.5-0.8 元 |
| 角色设计（Seedream） | 每角色 8 张（4 视图+4 表情） | ~0.1 元/张 | ~{角色数}×0.8 元 |
| 故事板（Seedream） | 每集 8 帧（4 段×2 帧） | ~0.1 元/张 | ~{总集数}×0.8 元 |
| 视频生成（Seedance） | 每集 4 段 | ~2-5 元/段（按时长和分辨率） | ~{总集数}×8-20 元 |
| TTS 音色（可选） | 每角色 1 次 | ~0.01 元/次 | 可忽略 |

> ⚠️ 以上为粗估，实际费用取决于火山方舟的计费规则和模型版本。Seedance 视频生成是主要成本项。
> 20 集项目的总 API 成本大约在 200-500 元范围（不含重试和精修）。

## 验收

### 宏观层面
- [ ] 主题具体可争论（不是空洞口号）
- [ ] 每集有明确信息增量和主题推进
- [ ] 悬念钩子足够强
- [ ] 人物 Want ≠ Need 成立
- [ ] 人物弧有说服力
- [ ] 角色关系图产生天然戏剧冲突
- [ ] 每个角色有语言指纹

### 跨集逻辑一致性检查（🆕）

> ⚠️ AI 生成的大纲最容易出现的问题是：A 集埋的信息在 B 集引用时主语搞错、同一事件在不同集的描述矛盾、角色知道了不该知道的信息。以下逐项检查。

- [ ] **信息归属正确**：每条"信息揭露"的主语明确——是观众知道了，还是角色 A 知道了，还是角色 B 知道了？不能混淆
- [ ] **角色认知边界**：每集中角色的行为是否只基于该角色当前已知的信息？（角色不能使用观众知道但角色不知道的信息）
- [ ] **身份/人物指代一致**：涉及多个人物时（如前任、跑掉的新娘、女主），每集对同一人物的指代是否一致？不能 A 集说"新娘"B 集说"前任"实际指同一个人但读者分不清
- [ ] **因果链完整**：每个"悬念钩子"在后续集中是否有对应的解答或推进？不能埋了钩子后面忘了
- [ ] **情感锚点建立**：后续集中有重要情感意义的道具/台词/动作，是否在前面集中建立了铺垫？（如第 13 集留下戒指，第 7 集必须先建立戒指的情感意义）
- [ ] **时间线无矛盾**：事件发生的先后顺序是否自洽？（如角色不能在 A 集"还不知道"某事，B 集突然"早就知道了"）
- [ ] **动机连贯**：反派/配角的行为动机在整季中是否一致？不能 A 集为了爱 B 集为了钱，除非有明确的转折点

### 骨架与叙事状态
- [ ] 80 段骨架覆盖全部集数，每段一句话（🆕）
- [ ] 骨架中信息揭露节奏均匀，无连续 3 段以上空转（🆕）
- [ ] 分集大纲已升级，含信息揭露 + 角色弧位置列（🆕）
- [ ] narrative-state.md 已初始化，未揭露信息已从骨架提取（🆕）

- [ ] 用户已确认

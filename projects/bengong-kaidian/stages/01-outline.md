# 阶段 01：整季叙事大纲

> 工具：DeepSeek（纯文本推理，不需要图片/视频模型）

## 前置条件

- 项目已初始化，`STATE.md` 存在

## 输入

- 用户偏好（题材、角色方向、画风、集数、目标受众）

## 流程

### 步骤 0：创意选题（如用户无明确方向）

> 用户可能只有模糊偏好（"甜宠"、"女性向"），没有具体故事。此步骤通过"创意漏斗"帮用户从模糊偏好收敛到具体题材。
> 如果用户已有明确故事方向，跳过此步骤直接进入步骤 1。

**创意漏斗流程**：

```
第 1 轮：类型推荐
  → 根据用户偏好（受众、题材方向），推荐 3-5 个爆款类型
  → 用户选定一个类型（如"霸总甜宠"）

第 2 轮：题材细化
  → 在选定类型下，给出 5-7 个具体题材方案
  → 每个方案包含：一句话概念 + 核心冲突 + 主角数 + 集数
  → 用户选定一个题材

第 3 轮：确认参数
  → 确认画风、对话呈现、目标受众等创作参数
  → 可由 AI 根据题材和受众推荐默认值
```

**类型推荐参考维度**：
- 目标受众的消费习惯（女频/男频/通吃）
- 当前市场验证过的爆款类型（重生复仇、霸总甜宠、悬疑反转、战神归来等）
- 适合 60 秒 × 多集格式的叙事结构（每集需要悬念钩子）
- AI 漫剧的视觉优势（场景切换少的题材比场景密集的题材更容易保持一致性）

### 步骤 1：收集创意输入 + 画风细化（合并）

**创意输入**：
- 故事题材（玄幻/都市/悬疑/甜宠/其他）
- 核心角色数量和基本设定方向
- 画风偏好（日漫/韩漫/国漫/写实/插画）
- 对话呈现风格：字幕模式 / 气泡台词模式 / 混合
- 每集时长 → **项目初始化时已通过 `--duration` 设定**，见 STATE.md 的"每集目标时长"字段。此处仅确认是否需要调整
- 总集数 → **项目初始化时已通过 `--episodes` 设定**
- 目标受众

**画风细化**（强烈推荐）：

> "日漫"不是一种风格——90 年代赛璐珞和 2010s 京都动画的视觉基因完全不同。细化画风大幅降低后续迭代次数。

| 维度 | 可选方向 | 示例 |
|------|---------|------|
| 年代参照 | 80s赛璐珞 / 90s经典 / 2000s数字 / 2010s现代 / 当代 | 2010s 现代数字动画 |
| 作品/工作室参照 | 京都动画风 / 新海诚风 / 吉卜力风 / MAPPA 风 | 新海诚风——极致光影、照片级背景 |
| 线条特征 | 粗重外轮廓 / 纤细均一 / 有笔触变化 / 无线条厚涂 | 纤细均一、干净利落 |
| 上色方式 | 硬边平涂 / 软边渐变 / 厚涂 / 水彩渗化 | 软边渐变、柔和过渡 |
| 阴影风格 | 分层色块式 / 柔和模糊 / 硬边阴影 / 网点纸 | 分层色块式、2-3 层明暗 |

将选定特征写入 `01-scripts/visual-style-spec.md`。

### 步骤 2：生成完整叙事蓝图（大纲 + 角色圣经 + 骨架一次性产出）

> ⚠️ **大纲、角色圣经、骨架三者有紧耦合关系**：角色圣经的 Want/Need 决定人物弧，人物弧驱动分集结构，分集结构展开为分集骨架。三者必须一起审核，不能分开通过。
>
> **产出策略**：三部分在 AI 内部按顺序生成，但作为**单次交付物**一次性提交用户审核。

#### 2a：`01-scripts/series-bible.md`

```markdown
# {剧名} 整季大纲

## 主题
具体、可争论。不是"爱与正义"，而是"偏执的复仇最终吞噬复仇者本身"。

## 中心戏剧问题
驱动全剧的核心问题。每集都在回答这个问题的一小部分。
例："林小夜能在找到凶手的同时不失去自己的人性吗？"

## 系列大弧
主角从{起点}到{终点}的完整旅程，要体现主题。

## 人物弧
详见 `character-bible.md`。此处仅列出各角色的弧线摘要：

### {角色A}
- 起始状态：{性格、处境}
- 成长路径：{遭遇什么、如何改变}
- 终局状态：{性格、处境}
（Want/Need/Flaw/Ghost 详见角色圣经，不在此重复）

## 角色关系图
标注：信赖/敌对/暗恋/利用/保护/竞争/师徒等

## 分集大纲（初版表格，步骤 3 全局扫描后补全"信息揭露"和"角色弧位置"列）
| 集数 | 标题 | 核心事件 | 悬念钩子 | 主题推进 |
|------|------|---------|----------|---------|
| 01 | ... | ... | ... | ... |

## 冲突主线
- A 故事线（主线）：...
- B 故事线（支线/感情线）：...
```

#### 2b：`01-scripts/character-bible.md`

```markdown
## {角色名}
### 外在
- 外貌：{描述}
- 服装：{描述}
- 标志性动作/物品：{咬嘴唇/转戒指/永远戴着帽子}

### 内在（四维）
- **Want**：{表面欲望}
- **Need**：{深层需求。关键：Want ≠ Need}
- **Flaw**：{核心缺陷/错误信念}
- **Ghost**：{创伤事件}

### 语言指纹
- 节奏：{快速/缓慢/时快时慢}
- 句长：{短句/长句/混合}
- 态度：{直接/迂回/讽刺/自我贬低/居高临下}
- 词汇特征：{文雅/粗俗/专业术语/网络用语}
- 口头禅或习惯句式：{...}
- 对白时的小动作：{摸耳朵/看地面/手不停比划}
```

#### 2c：`01-scripts/episode-skeleton.md`

> ⚠️ 这是全局叙事骨架。每集拆为 3-5 beat（按剧情复杂度浮动，不固定），每 beat 一句话，是后续逐集展开剧本的基础。
>
> 没有骨架就展开剧本 = 盲写，到第 10 集必然散架。

**beat 数量规则**：

| beat 数 | 适用场景 |
|--------|---------|
| 3 beat | 紧凑单场景集（纯对峙、纯告白、单一空间内完整事件） |
| 4 beat | 默认选择（经典微三幕结构） |
| 5 beat | 复杂多场景集（含转场、多视角、双线并进） |

> 硬约束：上限 5 beat（超过则拆分），下限 3 beat（少于则剧情单薄）。
> 每集标题后用 `（N beat）` 声明本集 beat 数量。

**编写方式**：分批生成，每次 5 集，每批由 AI 完成后做自检（质量检查通过），作为下一批的上下文。**不向用户分批确认**——全部集数骨架写完后，整体交付。

```
第 1 批：EP01-EP05（AI 自检通过后进入第 2 批）
第 2 批：EP06-EP10（带 EP01-05 骨架作为上下文）
第 3 批：EP11-EP15
第 4 批：EP16-EP20
```

> ⚠️ "每批自检"≠"每批用户确认"。分批的目的是控制 AI 单次生成长度，避免后半段质量下降。用户审核在所有 batch 完成后一次性进行。

**格式**：

```markdown
# {剧名} 分集骨架

## EP01: {集标题}（4 beat）
- beat1: {一句话，只写发生什么}
- beat2: {一句话}
- beat3: {一句话}
- beat4: {一句话} — 悬念：{钩子}

## EP02: {集标题}（3 beat）
- beat1: ...
- beat2: ...
- beat3: ... — 悬念：...

## EP03: {集标题}（5 beat）
- beat1: ...
- beat2: ...
- beat3: ...
- beat4: ...
- beat5: ... — 悬念：...
```

**字数规则**：
- 每 beat 一句话能说清楚即可，不设硬上限
- 一般在 20-50 字内；3-beat 集的单个 beat 可以稍长（承担信息更多），5-beat 集的 beat 可以更短
- 只写"发生什么"，不写镜头语言

**每批 AI 自检**（每批向下一批推进前）：
- 信息揭露节奏是否均匀（有没有连续 3 beat 以上在"聊天"而无新信息）
- 悬念钩子是否每集最后一 beat 都有
- 角色弧线是否平滑推进（对照 series-bible 的人物弧）
- 有没有重复的冲突模式（连续多集用同一种冲突类型）
- beat 数量选择是否合理（3/4/5 的依据是否与剧情复杂度匹配）

自检不通过则重写本批，不向下推进。

### 步骤 3：全局逻辑一致性扫描 + 初始化 narrative-state

> ⚠️ 分批生成容易漏掉跨批次的逻辑问题（如第 2 集埋的细节在第 15 集使用）。全部 beat 写完后，做一次完整的全局扫描。扫描过程中同时提取信息，产出 narrative-state.md 初始版和分集大纲补充列。

**逻辑检查**（逐条过）：
- [ ] **信息归属正确**：每条信息揭露的主语明确——是观众知道了，还是角色 A 知道了？不能混淆
- [ ] **角色认知边界**：每集中角色的行为是否只基于该角色当前已知的信息？
- [ ] **身份/人物指代一致**：涉及多个人物时，每集对同一人物的指代是否一致？
- [ ] **因果链完整**：列出所有悬念钩子，逐条检查后续是否有解答或推进。无解答的标记为遗漏
- [ ] **情感锚点闭环**：列出所有道具/台词/动作锚点，检查首次建立→中间强化→最终回收的完整链路
- [ ] **时间线无矛盾**：事件发生的先后顺序是否自洽？
- [ ] **动机连贯**：反派/配角的行为动机在整季中是否一致？
- [ ] **配角线收束**：所有在前半段出场的配角，在后半段是否有回响或收束？

**同步产出**（扫描过程中一并完成）：
1. **补充分集大纲**：在 `series-bible.md` 的分集大纲表格中补充"信息揭露"和"角色弧位置"列
2. **初始化 narrative-state.md**：从骨架中提取未揭露信息计划表、伏笔清单，初始化角色状态（格式见 `_convention.md` 的 narrative-state.md 规范）

### 步骤 4：整季成本估算

> 在用户最终确认前，提供 API 调用成本的估算区间。

**成本公式**（按 {总集数} 集、每集平均 4 beat × 约 2 seg/beat 估算）：

| 阶段 | 调用次数 | 单次成本（约） | 小计 |
|------|---------|--------------|------|
| Moodboard（Seedream） | 5-8 张 | ~0.1 元/张 | ~0.5-0.8 元 |
| 场景库（Seedream） | 每场景 2 张 | ~0.1 元/张 | ~{场景数}×0.2 元 |
| 角色设计（Seedream） | 每角色 2 张 | ~0.1 元/张 | ~{角色数}×0.2 元 |
| 视频生成（Seedance） | 每集 6-10 seg | ~2-5 元/段 | ~{总集数}×15-40 元 |
| TTS 音色 | 每角色 1 次 | ~0.01 元/次 | 可忽略 |

**含迭代与审计的现实成本**：

| 场景 | 系数 | 理由 |
|------|------|------|
| 基础成本 | ×1.0 | 一次通过 |
| 含迭代（30-50% 段需重跑） | ×1.4 | 正常项目 |
| 含跨集审计发现漂移 | ×1.55 | 第 5/10/15 集的重做 |

> 20 集项目**基础 API 成本约 300-800 元，含迭代与审计后约 500-1200 元**。
> 该估算由 AI 根据大纲复杂度粗算，实际费用取决于火山方舟当期计费。

### 步骤 5：用户一次性确认（合并确认）

> ⚠️ 以前大纲/角色/骨架分三次确认。现改为**一次性交付**——三者互相依赖，分开确认容易出现"角色改了但骨架没跟着改"。

提交以下全部产出物：
1. `series-bible.md`（含分集大纲升级后的完整表格）
2. `character-bible.md`
3. `episode-skeleton.md`（全部集骨架，每集标注 beat 数量）
4. `narrative-state.md`（初始版）
5. 成本估算区间

用户确认后进入阶段 02。

## 验收

### 宏观层面
- [ ] 主题具体可争论（不是空洞口号）
- [ ] 每集有明确信息增量和主题推进
- [ ] 悬念钩子足够强
- [ ] 人物 Want ≠ Need 成立
- [ ] 人物弧有说服力
- [ ] 角色关系图产生天然戏剧冲突
- [ ] 每个角色有语言指纹

### 跨集逻辑一致性检查

- [ ] **信息归属正确**：每条"信息揭露"的主语明确
- [ ] **角色认知边界**：每集中角色的行为只基于已知信息
- [ ] **身份/人物指代一致**：同一人物在各集中指代一致
- [ ] **因果链完整**：每个悬念钩子在后续集有解答或推进
- [ ] **情感锚点建立**：重要道具/台词/动作有前置铺垫
- [ ] **时间线无矛盾**：事件先后顺序自洽
- [ ] **动机连贯**：反派/配角动机在全季一致
- [ ] **配角线收束**：前半段配角在后半段有回响

### 骨架与叙事状态
- [ ] 分集骨架覆盖全部集数，每集标注 beat 数量（3/4/5）
- [ ] 每 beat 一句话描述清楚（不设硬字数上限）
- [ ] 每集最后一 beat 都有悬念钩子
- [ ] 骨架中信息揭露节奏均匀，无连续 3 beat 以上空转
- [ ] 分集大纲已升级，含信息揭露 + 角色弧位置列
- [ ] narrative-state.md 已初始化

### 最终确认
- [ ] 用户已一次性确认全部 5 项产出
- [ ] 成本估算已告知用户
